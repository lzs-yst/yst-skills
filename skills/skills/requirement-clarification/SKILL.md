# 需求澄清

## 目标

在开发前或需求讨论中，通过结构化提问补全信息，输出可执行的澄清结果（便于写 PRD 或直接开发）。

## 流程

1. **理解原始需求**：用一两句话复述用户描述的需求，确认没有曲解。
2. **按维度澄清**：针对缺失信息，从下面维度择要提问（不必一次问完，可多轮）。
3. **汇总输出**：将澄清结果整理成结构化摘要。

## 澄清维度与示例问题

| 维度 | 示例问题 |
|------|----------|
| 用户与场景 | 谁会用？在什么场景下用？要解决什么具体问题？ |
| 功能边界 | 必须做/不做/暂不做的范围？与现有功能的关系？ |
| 验收标准 | 怎样算做完？有无可量化的成功指标？ |
| 约束与优先级 | 时间/资源/依赖限制？MVP 与后续迭代的划分？ |
| 异常与边界 | 失败、无数据、权限不足时如何表现？ |

只问对当前需求有影响的点，避免泛泛而谈。

## 输出模板

澄清完成后，用以下结构输出（可只保留有结论的项）：

```markdown
# 需求澄清摘要：[需求名称]

## 背景与目标
[一两句：谁、什么场景、解决什么问题]

## 范围
- 在范围内：[…]
- 不在范围内 / 暂不做：[…]

## 验收标准
- [可检查的完成条件 1]
- [可检查的完成条件 2]

## 约束与优先级
- 约束：[时间/依赖/资源等]
- MVP：[本期必做]
- 后续：[可迭代项]

## 待定 / 风险
- [未决项或需后续确认的点]
```

## 注意

- 若用户已提供较完整描述，优先补全空白维度，而非重复提问。
- 用户明确说「先不澄清」或「先按现状做」时，可只做轻量复述与范围确认，不强制走完所有维度。
